# Lawful basis record

STUB — replace before requesting LIVE-OPEN

This file is a scaffold — replace it with an app-specific record before requesting LIVE-OPEN. The LIVE-OPEN gate checks that this file exists, is at least 500 characters, and contains the required heading below; it does NOT check that the content is adequate. Adequacy is the CEO's judgement at the `live_open_confirm` gate.

## Lawful basis

For each processing purpose the app carries, name the lawful basis and record enough evidence to defend the choice.

| Purpose | Lawful basis | Evidence / justification |
|---|---|---|
| _e.g. account creation and authentication_ | Contract | The user cannot use the app without an account; account creation is the pre-contractual step. |
| _e.g. analytics on usage_ | Legitimate interests | Balancing test recorded below; the interest is product improvement and users can opt out. |
| _e.g. marketing email_ | Consent | Opt-in checkbox at signup; consent record stored per user; withdrawal via link in every email. |

## Consent records

If any purpose uses consent, name where consent is recorded (which column in which table), what the consent statement is, and how withdrawal works.

## Legitimate-interest balancing tests

For every purpose relying on legitimate interests, record: the interest, the necessity (why nothing less intrusive works), and the balancing test (why the interest is not overridden by the data subject's rights). One paragraph each is fine.

## Special-category data

If any special-category data is processed (health, ethnicity, religion, biometrics, etc.), name the Article 9 condition (explicit consent, employment, vital interests, …) and record the additional safeguards.

## Children

If the app is directed at, or knowingly used by, children under the relevant age of consent, record the parental-consent mechanism and how age is verified.

## Changes

Record when this basis record was last reviewed. Review whenever the app adds a new processing purpose.
